# Phase 8 — Onboarding (Execution Plan)

## Objective

Ship a 14-screen onboarding flow that collects personalization data (persona, pain points, sleep/wake, energy peak, hard constraints, freeform context), persists it to `user_preferences` mid-flow, requests notification permission, and routes the user into the main app. Define an AI prompt helper that consumes the new fields (wiring into the Edge Function is a Phase 6 follow-up).

## Tasks

### Task 1 — Migration 004

Write `supabase/migrations/004_add_onboarding_fields.sql`. Add 5 columns with CHECK constraints. RLS unchanged. Comment each new column. Includes a default of `false` for `onboarding_completed`. Migration must be idempotent (`IF NOT EXISTS`).

### Task 2 — Type definitions

Extend `src/types/userPreferences.ts` with the 5 new fields. Create `src/types/onboarding.ts` exporting:
- `OnboardingData` — the in-memory shape (all collected fields)
- `Persona`, `ProductivityPeak`, `PainPoint`, `HardConstraint` string-literal unions
- Enum value arrays for UI consumption (e.g., `PERSONA_OPTIONS`)

### Task 3 — Constants

Three constants files. All hardcoded data, no DB:
- `onboarding-questions.ts` — radio/checkbox options with labels
- `onboarding-reviews.ts` — `Record<Persona, { name, quote, rating, avatarInitials }>`
- `onboarding-stats.ts` — `{ completionRate: [42, 78], replanningMinutes: [75, 8] }`

### Task 4 — In-memory onboarding state

`src/features/onboarding/state.ts`. Zustand store with `OnboardingData` and a `setField` helper. Reset on app boot. Exposed via a hook `useOnboardingState()`. **No persistence layer** — the store is the source of truth during the flow.

### Task 5 — Save API

`src/features/onboarding/api.ts`:
- `saveOnboardingData(data: OnboardingData): Promise<void>` — single UPSERT
- Uses existing Supabase client
- Throws on error; caller handles UI
- TanStack Query mutation not needed here (this is a one-shot, not a query)

### Task 6 — Routing hook

`src/hooks/useOnboardingStatus.ts`:
- Calls existing pattern of `useUserPreferences` (create if missing) to read `onboarding_completed`
- Returns `{ isLoading, hasCompletedOnboarding }`

`src/features/onboarding/hooks/useOnboardingRouting.ts`:
- Combines auth state + onboarding status
- Returns `{ isLoading, shouldShowAuth, shouldShowOnboarding, shouldShowTabs }`

## Notes

- The 12s processing theatre is a UX trick, not a real constraint. The save can be instant, the user just stares at checklist animations for 12s. If the save fails, we never advance — better to be stuck on a loading screen than to land in `(tabs)` with no data.
- Hard constraints mutual exclusion is enforced in the UI. The DB CHECK constraint allows any combination; if anyone writes `['none', 'work_hours']` directly, the AI prompt helper can handle it (treat `none` as authoritative). Document this in `ai-prompt.ts`.
- Schedule preview is static. If the real AI ever produces a personalized 1-day sample on demand cheaply, this can swap to a real call. Not v1.
- Existing `user_preferences` rows will all be treated as "needs onboarding." This is intentional — no real user has completed onboarding yet, so no backfill is needed.
- "Get Started" is the only CTA on the Hook. No secondary action. No "Learn more" link. The design is intentional: one path forward.
