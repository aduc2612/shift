# Phase 3 — Sentry Setup

## Objective

Add Sentry error monitoring, tracing, session replay, profiling, and logging to Shift. Catch JS + native crashes early, get source maps for production builds, and enable performance visibility from day one.

## Context

- Phase 1 & 2 complete: Expo project, folder structure, theme, Supabase client, auth, tab navigation all working
- Branch: `feat/phase-3-sentry-setup`
- No `metro.config.js` exists yet — need to create one
- `app.json` has plugins: `expo-router`, `expo-notifications`, `expo-splash-screen`, `expo-secure-store`
- Root layout at `src/app/_layout.tsx` uses `SafeAreaProvider` > `ThemeProvider` > `RootNavigator`
- `.env` uses `@env` pattern currently — migrating to `process.env.EXPO_PUBLIC_*` (Expo's recommended convention)
- Reference implementation: `C:\Ord` (same Expo + Sentry setup)

## Decision: Environment Variables

Migrating from `@env` (react-native-dotenv) to `process.env.EXPO_PUBLIC_*` (Expo's built-in convention).

Why:
- Expo recommends `EXPO_PUBLIC_*` prefix for client env vars
- No extra babel plugin needed
- Works consistently across Expo tooling (EAS, Metro, etc.)

Existing vars to migrate:
- `SUPABASE_URL` → `EXPO_PUBLIC_SUPABASE_URL`
- `SUPABASE_ANON_KEY` → `EXPO_PUBLIC_SUPABASE_ANON_KEY`
- `GOOGLE_WEB_CLIENT_ID` → `EXPO_PUBLIC_GOOGLE_WEB_CLIENT_ID`

New vars:
- `EXPO_PUBLIC_SENTRY_DSN`
- `SENTRY_AUTH_TOKEN` (build-time only, not client-exposed)

**5b. Add `Sentry.init()` before the component definitions:**
```ts
Sentry.init({
  dsn: process.env.EXPO_PUBLIC_SENTRY_DSN,
  sendDefaultPii: true,

  // Tracing — 100% in dev, 20% in production
  tracesSampleRate: __DEV__ ? 1.0 : 0.2,

  // Profiling — lower in production
  profilesSampleRate: __DEV__ ? 1.0 : 0.1,

  // Session Replay — always on error, 10% sessions in prod
  replaysOnErrorSampleRate: 1.0,
  replaysSessionSampleRate: __DEV__ ? 1.0 : 0.1,

  // Logging
  enableLogs: true,

  // Mobile Replay integration
  integrations: [
    Sentry.mobileReplayIntegration({
      maskAllText: true,
      maskAllImages: true,
    }),
  ],

  // Disable native frames tracking in Expo Go (not supported)
  enableNativeFramesTracking: !isRunningInExpoGo(),

  environment: __DEV__ ? "development" : "production",
  debug: __DEV__,
});
```

**5c. Wrap the default export:**
```ts
export default Sentry.wrap(RootLayout);
```

## Notes

- **Expo Go limitation:** `enableNativeFramesTracking` must be `false` in Expo Go — handled by `!isRunningInExpoGo()`
- **Source map uploads:** Config plugin handles `sentry.properties` generation during prebuild. `SENTRY_AUTH_TOKEN` in `.env` is used at build time, not bundled in client.
- **Production sampling:** `tracesSampleRate: 0.2`, `profilesSampleRate: 0.1`, `replaysSessionSampleRate: 0.1` are reasonable defaults for production. Adjust after launch based on volume.
- **No `Sentry.wrap()` for Expo Router:** Expo Router handles navigation tracking automatically — no `Sentry.NavigationContainer` needed.
- **Privacy:** `sendDefaultPii: true` sends IP addresses and user IDs to Sentry. Ensure this aligns with privacy policy.
- **`SENTRY_AUTH_TOKEN` in `.env`:** This is a build-time variable used by the Expo config plugin for source map uploads. It's NOT bundled into the client binary. Safe to keep in `.env`.

## Manual Steps (User)

Before or after implementation:

1. Go to [sentry.io](https://sentry.io/) → create project → select React Native
2. Copy DSN → add to `.env` as `EXPO_PUBLIC_SENTRY_DSN`
3. Go to Settings → Auth Tokens → create token with `org:read`, `project:releases`, `event:write` scopes
4. Add token to `.env` as `SENTRY_AUTH_TOKEN`
5. For EAS builds: `eas secret:create --scope project --name SENTRY_AUTH_TOKEN --value <token>`
